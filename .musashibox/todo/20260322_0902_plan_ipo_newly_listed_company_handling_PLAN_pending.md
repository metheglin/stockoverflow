# plan_ipo_newly_listed_company_handling

## 概要

新規上場（IPO）企業のデータ蓄積ルール、スクリーニングでの取り扱い、上場日の管理方法を設計する。

## 背景・目的

現在の仕組みでは、SyncCompaniesJob が JQUANTS の listed info から企業マスターを同期するが、**いつ上場したか**の情報を保持していない。また、IPO企業は以下の特殊性がある：

1. **過去データの不足**: 上場後1〜2年は consecutive_revenue_growth 等の指標が限定的
2. **スクリーニングの公平性**: 「6期連続増収増益」のスクリーニングにIPO後2年目の企業が含まれることは不適切（データ不足により候補に上がらないだけで、実質的に高成長企業を見逃す可能性）
3. **IPO企業の追跡ニーズ**: 新興成長企業を早期に発見するため、IPO企業だけを一覧・追跡する機能は有用
4. **上場前のデータ**: 一部のIPO企業はEDINETに上場前の有価証券届出書を提出しており、そこから過去データを取得できる可能性がある

- `new_company_automatic_data_backfill` は新規追加企業のデータバックフィルに焦点。IPO固有の問題（上場日管理、スクリーニング除外ルール等）は含まれていない

## 設計すべき項目

### 1. 上場日の管理

- companies テーブルへの `listed_on` (date) カラム追加の検討
- JQUANTS の listed info から上場日を取得可能か調査
- 取得不可の場合、SyncCompaniesJob で「初めて出現した日」を listed_on として記録する代替策

### 2. IPO企業の判定ルール

- 上場後N年以内をIPO企業とみなす基準の設計
- 「データ蓄積期間」（上場日からの経過年数ではなく、実際のfinancial_valuesの期数）に基づく判定

### 3. スクリーニングでの取り扱い

- consecutive growth 系のスクリーニングにおいて、IPO企業を:
  - (a) 除外する
  - (b) 別枠で表示する（「データ蓄積中」ラベル付与）
  - (c) 「N期連続」の閾値を動的に調整する
- どのアプローチが適切か設計

### 4. IPO企業専用のスクリーニング

- 上場後N年以内の企業一覧
- 上場後の成長率ランキング
- IPO初年度からの業績推移

### 5. 上場前データの取得可能性

- EDINETの有価証券届出書（新規公開時）から取得できるデータの調査
- 取得できる場合の FinancialReport / FinancialValue への格納方法
