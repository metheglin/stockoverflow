# データ量増加予測とリテンション戦略の計画

## 概要

本プロジェクトはSQLiteを採用しており、日次株価データ・四半期決算データ・算出指標が継続的に蓄積される。
100以上の分析系TODOを実装する前に、想定データ量でSQLiteが分析クエリに耐えうるか検証し、
必要に応じてリテンション（保持）ポリシーやアーカイブ戦略を策定する。

## 背景

- 上場企業数: 約3,800社
- daily_quotes: 約3,800社 x 250営業日 = 年間約95万行
  - 5年分で約475万行、10年分で約950万行
- financial_values: 約3,800社 x 4四半期 x 2スコープ = 年間約30,400行
- financial_metrics: financial_valuesと同等
- SQLiteは数千万行でも動作するが、分析クエリ（集計・ランキング・複数テーブルJOIN）のパフォーマンスは検証が必要
- `improve_sqlite_query_performance` はクエリ最適化、`plan_sqlite_backup_restore_strategy` はバックアップに焦点。本タスクはデータ量そのものの長期見通しを扱う

## 計画で検討すべき事項

### 1. データ量の定量的予測

- テーブルごとに1年/3年/5年/10年のレコード数を算出
- レコードサイズ（平均バイト数）を考慮したDB容量の見積もり
- インデックスを含めた総容量

### 2. クエリパフォーマンスの検証方針

- コアユースケース3つのクエリを想定データ量で検証する方法
- ダミーデータ生成によるストレステスト計画
- SQLiteのEXPLAIN QUERY PLANを活用したボトルネック特定

### 3. リテンションポリシーの検討

- daily_quotes: N年以上前のデータを週次/月次に集約するか
- 全データ保持 vs 集約保持のトレードオフ
- テクニカル指標計算に必要な最小データ粒度

### 4. アーカイブ戦略

- 古いデータの別DBファイルへの分離（SQLiteのATTACH DATABASE活用）
- 読み取り専用のアーカイブDBと書き込み用のアクティブDBの分離
- バックアップとの統合

### 5. 成果物

- データ量予測表
- リテンションポリシー案
- 必要に応じてアーカイブ実装のDEVELOP TODO
