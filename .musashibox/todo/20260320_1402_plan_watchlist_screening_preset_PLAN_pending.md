# ウォッチリスト・スクリーニングプリセット管理の設計

## 概要

スクリーニング条件の保存・再実行と、注目企業のウォッチリスト管理機能の設計をおこなう。
プロジェクトの中核ユースケース（条件による企業の絞り込み・一覧）を反復的に利用可能にするための機能設計。

## 背景

- `dev_analysis_query_layer` はクエリの「仕組み」を提供するが、ユーザーが定義した条件の「永続化と管理」はスコープ外
- プロジェクトのユースケース（6期連続増収増益の一覧、CF転換企業の一覧など）は繰り返し実行されるものであり、毎回条件を組み立てるのは非効率
- 条件の変化を追跡することで「前回のスクリーニング結果との差分」も把握可能になる

## 設計対象

### スクリーニングプリセット

- スクリーニング条件を名前付きで保存するテーブル設計
  - 条件はJSON型で柔軟に表現（フィルタ条件、ソート順、表示項目）
  - プリセットの種類（ビルトイン / ユーザー定義）
- ビルトインプリセットの候補
  - 連続増収増益（N期以上、増収率ソート）
  - FCFプラス転換
  - 高ROE + 低PBR（バリュー）
  - 高成長 + 高収益（グロース）
- プリセットの実行結果の一時保存とスナップショット比較

### ウォッチリスト

- 企業をグループ化して管理するテーブル設計
  - ウォッチリスト名、説明、企業リスト
  - 企業追加時のメモ（なぜ注目しているか）
- スクリーニング結果をウォッチリストに変換する機能

### 検討事項

- テーブル設計（screening_presets, watchlists, watchlist_items）
- JSON型の条件スキーマ定義
- analysis_query_layerとの連携インターフェース
- CLIからの操作方法（rake task / rails runner）
- 将来的なWeb UI対応を見据えたAPI設計
