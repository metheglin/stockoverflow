# SQLiteデータベースバックアップ・リストア戦略の設計

## 概要

長期にわたり蓄積される財務データを保護するため、SQLiteデータベースのバックアップ戦略を設計する。データ収集には大量のAPI呼び出しと時間を要するため、データベースの喪失は甚大な損害となる。

## 背景

- 本プロジェクトは個人用データベースとしてSQLiteを採用しており、データ収集にはJQUANTS/EDINET APIへの大量のリクエストが必要
- フルインポート（特にdaily_quotes）には数時間〜数日を要する可能性がある
- SQLiteは単一ファイルDB であるためバックアップが比較的容易だが、書き込み中のコピーはデータ破損のリスクがある
- 現時点でバックアップに関する仕組みが一切存在しない

## 設計すべき項目

### 1. バックアップ手法の選定

以下の方式を比較検討する:

#### 案A: SQLite Online Backup API (`sqlite3 .backup`)
- SQLite組み込みのバックアップ機能
- 書き込み中でも安全にバックアップ可能
- Railsから `ActiveRecord::Base.connection.raw_connection` 経由でアクセス可能

#### 案B: Litestream
- SQLiteのストリーミングレプリケーションツール
- WALファイルを継続的にS3等にレプリケート
- ポイントインタイムリカバリが可能
- ただし外部ツールへの依存が発生

#### 案C: ファイルコピー + WALチェックポイント
- `PRAGMA wal_checkpoint(TRUNCATE)` 後にファイルコピー
- 最もシンプルだがダウンタイムが発生する

### 2. バックアップスケジュール

- 日次バックアップの実行タイミング（全ジョブ完了後が理想）
- バックアップのローテーション・保持期間
- 保存先（ローカルディスク、外部ストレージ）

### 3. リストア手順

- バックアップからの復旧手順の文書化
- 復旧後のデータ整合性検証方法
- 部分的なデータ復旧（特定テーブルのみ等）の可否

### 4. データベースサイズの見積もり・監視

- 想定データ量に基づくDBファイルサイズの推移予測
  - companies: ~4,000レコード（数MB）
  - financial_values: ~80,000レコード（4,000社 x 20期、数十MB）
  - financial_metrics: ~80,000レコード（同上）
  - daily_quotes: ~5,000,000レコード（4,000社 x 5年 x 250日、数百MB〜1GB）
- DBサイズ監視のrakeタスクまたはヘルスチェック

### 5. WAL管理

- WALファイルの肥大化防止策
- 定期的なチェックポイント実行のタイミング
- `dev_job_scheduling` (20260310_1402) との連携
