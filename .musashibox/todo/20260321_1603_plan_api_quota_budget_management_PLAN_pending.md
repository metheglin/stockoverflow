# PLAN: API呼び出しクォータ・バジェット管理の設計

## 概要

EDINET APIおよびJQUANTS APIの呼び出し回数・頻度を管理し、レートリミットやクォータ超過を防ぐバジェット管理の仕組みを設計する。

## 背景・動機

- JQUANTS APIでは429（レートリミット）エラーが発生することが確認されており、`ImportJquantsFinancialDataJob` で対策済みだが、対処療法的なスリープ挿入に留まっている
- フルインポート時には数千社 x 複数エンドポイントの呼び出しが発生し、数時間〜数日にわたるAPI呼び出しとなる
- EDINET APIにはAPIキーごとの日次呼び出し上限が存在する可能性があり、大量の書類ダウンロードで上限に達するリスクがある
- 複数のジョブが並行して同じAPIを呼び出す場合、合計のレート超過が発生しうる（現状はSolid Queueの逐次実行に依存）
- APIの利用状況（何回呼び出したか、残りクォータはいくつか）を把握する手段がない

## 設計検討事項

### 1. API呼び出し統計の記録

各API呼び出しの回数と結果を記録する仕組み:

- 記録先: `ApplicationProperty` (kind: :api_usage) の data_json
- 記録内容:
  - 日付ごとのAPI呼び出し回数（エンドポイント別）
  - 成功/失敗/レートリミットの内訳
  - 最後の呼び出し時刻

### 2. レートリミット戦略

現在の個別ジョブでのスリープ定数をより適応的な仕組みに進化させる:

- **固定レート方式**（現状）: `SLEEP_BETWEEN_REQUESTS = 2` 等の固定スリープ
- **適応レート方式**（目標）: APIレスポンスヘッダーやエラー率に基づいてスリープ時間を動的に調整
- **トークンバケット方式**: 一定期間あたりの最大呼び出し回数を設定し、バケットが空になったら待機

### 3. クォータ予算の設定

- JQUANTS: 1日あたり / 1分あたりの最大呼び出し回数を設定可能にする
- EDINET: 1日あたりの最大書類ダウンロード数を設定可能にする
- 予算残量が少なくなった場合にジョブを graceful に停止し、翌日に再開可能にする

### 4. ジョブ間の調整

- 複数ジョブが同一APIを利用する場合のリソース競合の防止
- PipelineOrchestrationJob がジョブの逐次実行を保証する前提での設計
- 将来的に並列実行を許可する場合の拡張性

### 5. 監視と通知

- 日次のAPI利用量レポート（rakeタスク or ログ出力）
- クォータ上限の80%到達時のワーニングログ出力
- レートリミットエラーの頻度が閾値を超えた場合のアラート
