# PLAN: Webダッシュボード・企業ブラウザの設計

## 背景

現在、蓄積された財務データにアクセスする手段が存在しない。`config/routes.rb` にはヘルスチェックのみが定義されている。

`plan_web_api` (REST API設計) が計画済みだが、APIだけでは日常的にデータを閲覧・分析することが難しい。Gemfile上は Turbo-rails、Stimulus-rails、Propshaft が導入済みであるにもかかわらず、フロントエンドは一切構築されていない。

個人用データベースとして実用的に活用するため、ブラウザベースのダッシュボード・企業ブラウザを設計する。

## 設計すべき項目

### 1. 画面構成

以下の画面を設計すること:

#### ダッシュボード（トップページ）
- データ同期状況（最終同期日時、staleness状況）
- 注目指標サマリー（連続増収増益企業数、直近のトレンド転換企業数など）
- 直近の決算発表・開示情報

#### 企業一覧・スクリーニング画面
- 分析クエリレイヤーの `ScreeningQuery` をUIで操作する
- 業種・市場区分のフィルタ
- 指標に基づくフィルタ（ROE > 10%、連続増収3期以上 等）
- ソート切り替え
- ページネーション

#### 企業詳細画面
- 企業基本情報
- `FinancialTimelineQuery` の結果をテーブルおよびチャートで表示
- 主要指標の推移グラフ（売上・利益・ROE・マージン推移）
- セクター平均との比較（`SectorComparisonQuery` を利用）
- 株価チャート（`daily_quotes` を利用）

#### セクター一覧・詳細画面
- セクター別統計サマリー
- セクター内の企業ランキング

### 2. 技術スタック

以下の方針を検討すること:
- **Turbo + Stimulus**: 既に導入済みのRails標準スタック
- **チャート描画**: Chart.js等のJavaScriptライブラリの選定
- **テーブル表示**: ソート・フィルタ可能なテーブルUIの方針
- **レスポンシブ対応**: PC中心だが基本的なレスポンシブは確保

### 3. コントローラー・ルーティング設計

- REST APIエンドポイントとHTML表示エンドポイントの共存方法
- 認証（簡易的なBasic認証 or API Token）

### 4. 段階的な実装計画

全画面を一度に実装するのは非現実的。以下の優先順位で段階的に実装する計画を設計すること:
1. 企業一覧（最もシンプルかつ実用性が高い）
2. 企業詳細（データ分析のコア機能）
3. ダッシュボード
4. セクター画面

## 前提

- Web API設計（`plan_web_api`）が完了していること
- 分析クエリレイヤー（`dev_analysis_query_layer`）が実装済みであること
