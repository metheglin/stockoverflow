# PLAN: トレンド転換検出・フラグ付け機能の設計

## 背景

CLAUDE.mdの想定ユースケース3:
> （数年にわたって素晴らしい成果を出した）ある企業の業績が飛躍し始める直前にどのような決算・財務上の変化があったかを調べる

現在、`Company::FinancialTimelineQuery`（分析クエリレイヤーで設計済み）は特定企業の時系列データを返すが、そのデータのどこで「トレンドの転換」が起きたかは人間が目視で判断する必要がある。

企業数が多い中で「業績が飛躍し始める直前」のパターンを効率よく検出するには、トレンド転換を自動的に検出しフラグ付けする仕組みが必要。これにより「今まさにトレンドが転換しつつある企業」を抽出するスクリーニングも可能になる。

## 設計すべき項目

### 1. トレンド転換の定義

以下のようなパターンを「トレンド転換」として定義し、検出ロジックを設計する。

- **増収増益の開始**: `consecutive_revenue_growth` が 0 → 1 に転じた期（特に前期まで複数期減収だった場合は注目度高）
- **利益率の底打ち**: `operating_margin` が複数期連続低下後に反転上昇した期
- **フリーCFの黒字転換**: `free_cf_positive` が false → true に転じた期
- **ROEの反転上昇**: ROEが複数期連続低下後に反転した期
- **売上成長率の加速**: `revenue_yoy` が正で、かつ前期の `revenue_yoy` を上回った期（成長加速）
- **バリュエーション変化**: PER/PBRが業種平均対比で急変した期

### 2. 検出結果の保持方法

以下を検討する:
- `financial_metrics.data_json` 内に転換フラグを格納する方式
- 新たなEAVテーブル（company_properties的な）でイベントとして格納する方式
- 専用テーブル `trend_events` を設ける方式

それぞれの長所短所と、分析の柔軟性・検索性のバランスを考慮すること。

### 3. スクリーニングへの統合

検出されたトレンド転換をスクリーニング条件として利用する方法を設計する。
- 「直近1年以内にフリーCF黒字転換した企業」
- 「今期初めて連続増収増益2期目に入った企業」
- 「利益率が底打ち反転した企業」

### 4. CalculateFinancialMetricsJob との関連

既存のジョブに統合するか、別ジョブとして切り出すかを検討する。
