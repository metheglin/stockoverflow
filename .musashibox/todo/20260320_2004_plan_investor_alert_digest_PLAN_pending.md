# PLAN: 投資家向けアラート・ダイジェスト配信システムの設計

## 概要

蓄積されたデータ分析結果・財務イベント・スクリーニング結果の変化を、投資家が日常的に活用できる形式で集約・配信するシステムを設計する。

## 背景・動機

本プロジェクトの最終目標は「注目すべき企業や情報を一覧できるようなシステム」であるが、現在の設計はデータの蓄積・算出・クエリに集中しており、分析結果をユーザーに能動的に届ける仕組みが存在しない。

以下の分析基盤が整備されつつある:
- 財務指標のYoY成長率・連続成長期数の算出
- フリーCF転換検出
- スクリーニングクエリ（analysis_query_layer）
- 財務イベント検出（dev_financial_event_detection）
- トレンド分類（dev_metric_trend_classification）

これらの分析結果を「毎日確認したくなるダイジェスト」として集約し配信する仕組みがなければ、データベースとしての価値は発揮されても投資判断支援ツールとしての実用性に欠ける。

## 設計対象

### 1. ダイジェストの内容構成

#### デイリーダイジェスト（毎営業日生成）

- **新規データサマリー**: 当日インポートされた決算報告数、株価更新企業数
- **注目イベント一覧**: 財務イベント（streak_started, fcf_turned_positive等）を severity 順に表示
- **スクリーニング結果の変動**: 各プリセットスクリーニングの「新規該当企業」「脱落企業」
- **ウォッチリスト企業の変化**: ウォッチリスト登録企業の新規決算・指標変化

#### ウィークリーサマリー（週次生成）

- 週間の新規決算報告件数・メトリクス更新件数
- 週間で発生した重要イベントの集約
- スクリーニング結果の週次比較
- 市場全体のトレンドサマリー（セクター別平均成長率の変化等）

### 2. 配信チャネルの設計

優先度順に以下の配信方式を検討する:

#### Phase 1: CLIレポート生成（rake task / rails runner）
- `rake digest:daily` で当日のダイジェストをテキスト出力
- `rake digest:weekly` で週次サマリーを出力
- 出力形式: 構造化テキスト（Markdown互換）
- 最も実装コストが低く、cron連携で自動化可能

#### Phase 2: JSONエンドポイント
- `GET /api/v1/digest/daily` でJSON形式のダイジェストを返す
- Web UIやSlack Bot等の外部ツールから利用可能
- dev_web_api の一部として実装

#### Phase 3: Webhook通知（Slack / Discord）
- ダイジェスト生成後にWebhook URLにPOST
- 設定はApplicationPropertyまたはcredentialsで管理
- 重要イベント（severity: critical）は即座に通知、その他はダイジェストに集約

### 3. ダイジェスト生成のアーキテクチャ

#### DigestGeneratorJob

```
データフロー:
1. 前回ダイジェスト生成時刻以降の FinancialEvent を取得
2. 前回以降にインポートされた FinancialReport / FinancialValue を集計
3. 登録済みスクリーニングプリセットを実行し、前回結果とのdiffを算出
4. ウォッチリスト企業のメトリクス変化を検出
5. 上記を構造化データとして集約
6. 配信チャネルに応じた形式に変換して出力
```

#### ダイジェスト結果の永続化

- `application_properties` (kind: :digest) に最新ダイジェストの生成結果を保存
- 前回結果との比較のため、前回のスクリーニング結果スナップショットも保持
- 古いスナップショットは一定期間後に自動削除

### 4. 優先度付けとフィルタリング

ダイジェストの情報過多を防ぐため:
- イベントの severity に基づく優先度ソート
- ウォッチリスト登録企業のイベントを上位表示
- 時価総額・市場区分によるフィルタ（プライム市場のみ等）
- 1日のダイジェストに含めるイベント上限の設定

## 設計で検討すべき事項

- ダイジェスト生成の実行タイミング（CalculateFinancialMetricsJob完了後 or 独立スケジュール）
- スクリーニング結果スナップショットの保存量（ストレージ制約）
- 複数配信チャネルの同時利用時の設計
- ダイジェストのカスタマイズ性（どの情報を含めるかのユーザー設定）
- テスト方針（ジョブ実行テストは不要だが、ダイジェスト組み立てロジックはテスト対象）
