# スクリーニング条件の状態変化検出システム設計

## 概要

現在のスクリーニング（dev_analysis_query_layer で計画中）は「現時点でこの条件を満たす企業の一覧」を取得するものだが、本計画では「新たに条件を満たすようになった企業」「条件を満たさなくなった企業」の検出メカニズムを設計する。

## 背景・目的

プロジェクト目標の「注目すべき企業や情報を一覧できるようなシステム」を実現するうえで、静的なスクリーニング結果だけでなく、動的な変化を捉えることが重要。

例:
- ある企業が今回の決算で「6期連続増収増益」の条件を新たに達成した
- 営業CF + 投資CFの差がマイナスからプラスに転換した企業を検出した
- 直前まで高いROEを維持していた企業のROEが急落した

これらは「前回のスクリーニング結果」と「今回のスクリーニング結果」を比較することで検出できる。

## 設計検討事項

### アプローチ案

#### 案A: スナップショット比較方式
- スクリーニング結果をスナップショットとして保存する
- 次回実行時に前回スナップショットと差分比較
- 新規該当・脱落を検出
- 保存先: `application_properties` のEAVパターン or 専用テーブル

#### 案B: イベント駆動方式
- `CalculateFinancialMetricsJob` 実行時に、特定の閾値を超えたか判定
- 閾値超過イベントを記録するテーブルを作成
- よりリアルタイムだが、スクリーニング条件ごとにロジックが必要

#### 案C: ハイブリッド方式
- 重要な指標の閾値超過はイベント駆動で検出（案B）
- 複雑なスクリーニング条件の変化はスナップショット比較（案A）

### 検討すべき項目

1. **スクリーニング条件の定義方法**: ハードコードか、設定可能か
2. **通知・出力手段**: ログ出力、CSV出力、将来的にはWeb UI / メール
3. **検出頻度**: メトリクス計算ジョブと連動 or 独立スケジュール
4. **履歴保持**: いつどの企業がどの条件を達成/脱落したかの履歴
5. **初期スクリーニング条件の選定**: プロジェクト目標のユースケースから優先度を決定

### デフォルトで検出すべき状態変化候補

- 連続増収N期達成（N=3,5,6）
- 連続増益N期達成
- FCF（フリーキャッシュフロー）がマイナスからプラスに転換
- ROE が一定水準（8%, 10%, 15%）を超過/下回り
- 営業利益率が一定水準を超過/下回り
