# WORKLOG: トレンド転換検出・フラグ付け機能の設計

作業日時: 2026-03-24

## 作業概要

PLAN TODO `20260319_1401_plan_trend_turning_point_detection_PLAN_inprogress.md` に基づき、トレンド転換検出機能の詳細設計書を作成した。

## 考えたこと

### データ保持方式の選定

3つの方式を比較検討した:

1. **financial_metrics.data_json 内にフラグ格納** — スクリーニング用途にはSQLiteのJSON検索が非効率であり不採用
2. **EAVテーブル（company_properties的）** — 転換点は「エンティティの属性」ではなく「時系列上のイベント」であり、EAVとの相性が悪く不採用
3. **専用テーブル trend_turning_points** — スクリーニングのためのインデックス活用、時系列イベントとしての自然な構造から採用

### 検出パターンの設計

既存の `FinancialMetric` が保持するデータを活かし、以下の6パターンを定義:

- P1〜P3（優先度A）: 増収転換、フリーCF黒字転換、利益率底打ち — これらは `FinancialMetric` の固定カラムから直接検出可能で、最も実装が容易
- P4〜P5（優先度B）: ROE反転、成長加速 — P3と同様のロジックパターンで実装可能
- P6（優先度C）: バリュエーション急変 — `SectorMetric` への依存があり、実装順序として後回し

### ジョブの分離

`CalculateFinancialMetricsJob` への統合ではなく、別ジョブ `DetectTrendTurningPointsJob` として設計した理由:
- 転換点検出は3〜5期分の履歴が必要で、1期ずつ処理する既存ジョブとは処理パターンが異なる
- SectorMetric も参照するため、全指標算出完了後に実行すべき
- 責務を明確に分離することで、検出ロジックのテストや再実行が容易

### 関連TODOとの整合性

以下の既存TODOとの重複・競合を確認:
- `dev_financial_event_detection` — より汎用的なイベント検出テーブル。将来的な統合パスは残しつつ、本機能はトレンド転換に特化した独立実装とした
- `dev_metric_trend_classification` — トレンドの「方向」分類であり、本機能は転換の「瞬間」検出で異なる
- `dev_growth_acceleration_metrics` — 成長加速度データ。利用可能だが依存はしない設計

### 注目度（significance）の導入

単純な転換検出だけでなく、「長期低迷からの反転」と「1期だけの変動」を区別するため、3段階の注目度を設けた。これによりスクリーニング時に本当に注目すべき企業を優先できる。
