# データ量推計レポート

## 概要

本レポートでは、日本の上場会社数・決算回数・取引日数などの公開情報に基づき、実装済みJOBを実行した場合に各テーブルに登録されるレコード数を推計する。

---

## 前提条件

### 日本の上場企業数

東京証券取引所（TSE）の上場企業数（2025年末時点の概算）:

| 市場区分 | 企業数（概算） |
|---------|-------------|
| プライム | 約1,650社 |
| スタンダード | 約1,600社 |
| グロース | 約600社 |
| **TSE合計** | **約3,850〜3,950社** |

JQUANTS APIが返す上場銘柄一覧はTSE上場銘柄を対象としており、名証・福証・札証の独自上場銘柄（数十社程度）も一部含む場合がある。本推計では **約4,000社** を基準値として使用する。

### 決算期の分布

日本の上場企業の決算期分布:

| 決算月 | 比率（概算） |
|-------|-------------|
| 3月決算 | 約65〜70% |
| 12月決算 | 約10〜12% |
| 9月決算 | 約3〜4% |
| 6月決算 | 約2〜3% |
| その他 | 約15〜20% |

3月決算企業が圧倒的に多いため、有価証券報告書の提出は6月に集中し、四半期報告の提出は8月・11月・2月にピークが発生する。

### 四半期報告書制度の改正（2024年4月施行）

2024年4月以降、四半期報告書（Q1・Q3）の法定開示義務が廃止され、以下のように変更された:

| 書類 | 改正前 | 改正後 |
|-----|--------|--------|
| 有価証券報告書（通期） | 年1回（EDINET） | 年1回（EDINET）※変更なし |
| 四半期報告書（Q1, Q3） | 年2回（EDINET） | **廃止** |
| 半期報告書（Q2） | - | 年1回（EDINET）※四半期報告書Q2を置換 |
| 決算短信（Q1〜Q3, 通期） | 年4回（TDnet） | 年4回（TDnet）※変更なし |

この改正により:
- **EDINETから取得可能な報告書**: 年2回（通期 + 半期）に減少
- **JQUANTSの財務情報サマリー**: 決算短信ベースのため年4回（Q1, Q2, Q3, 通期）を維持

### 取引日数

東京証券取引所の年間取引日数:

| 項目 | 日数 |
|-----|------|
| 年間カレンダー日数 | 365日 |
| 土日 | 約104日 |
| 祝日（取引所休場日） | 約16〜18日 |
| 年末年始（12/31〜1/3） | 上記に含む |
| **年間取引日数** | **約243〜245日** |

本推計では **年間245取引日** を基準値として使用する。

### JQUANTSのデータ提供範囲

JQUANTSのプランによってデータ取得可能範囲が異なる:

| プラン | 株価データ | 財務データ |
|-------|-----------|-----------|
| Free | 直近12週間 | 直近2年間 |
| Light | 2012年以降 | 2017年以降 |
| Standard | 2012年以降 | 2017年以降 |
| Premium | 2012年以降 | 2017年以降 |

本推計では、有料プラン（Light以上）を前提とし、以下の期間を想定する:
- **株価データ**: 2020年1月1日〜現在（`ImportDailyQuotesJob` fullモードのデフォルト `from_date`）= 約6年分
- **財務データ**: 2017年〜現在 = 約8〜9年分

## テーブル別レコード数推計

### 1. companies（企業マスター）

**登録元JOB**: `SyncCompaniesJob`

| 項目 | 値 |
|-----|-----|
| JQUANTSの上場銘柄一覧 | 約4,000社 |
| EDINET由来の追加企業（非上場の有報提出企業等） | 最大数百社 |

`SyncCompaniesJob` はJQUANTS上場銘柄一覧から `securities_code` をキーにupsertする。`ImportEdinetDocumentsJob` は証券コード・EDINETコードで企業を検索し、見つからない場合のみ新規作成する（非上場の有報提出企業等）。

| シナリオ | 推計レコード数 |
|---------|-------------|
| JQUANTSのみ実行 | **約4,000件** |
| JQUANTS + EDINET実行 | **約4,000〜4,500件** |

### 2. financial_reports（決算報告書メタデータ）

**登録元JOB**: `ImportJquantsFinancialDataJob`, `ImportEdinetDocumentsJob`

JQUANTS由来とEDINET由来で **別々の `doc_id`** が採番されるため、同一決算期のデータでも別レコードとして登録される。

#### JQUANTS由来

doc_id形式: `JQ_{Code}_{FYEnd}_{PeriodType}`

| 項目 | 計算 | 件数 |
|-----|------|------|
| 1社あたりの年間報告数 | 4（通期 + Q1 + Q2 + Q3） | 4 |
| 対象企業数 | 約4,000社 | 4,000 |
| 年間合計 | 4,000 × 4 | 16,000/年 |
| 8年分（2017〜2024） | 16,000 × 8 | **約128,000件** |

ただし、過去に上場廃止した企業のデータも含まれるため、年によって対象企業数は変動する。また一部の企業はQ1・Q3データを提出しない場合がある。

#### EDINET由来

| 項目 | 計算 | 件数 |
|-----|------|------|
| 対象書類種別 | 有価証券報告書(120,130) + 四半期報告書(140,150) + 半期報告書(160,170) | - |
| 2024年3月以前: 1社あたり年間 | 4（通期 + Q1 + Q2 + Q3） | 4 |
| 2024年4月以降: 1社あたり年間 | 2（通期 + 半期） | 2 |
| 訂正報告書 | 全体の約5〜10% | +5〜10% |

EDINET由来のレコード数は、`ImportEdinetDocumentsJob` でスキャンする日付範囲に依存する。デフォルトは「最終同期日から昨日まで」であり、初回は30日前から。

| スキャン期間 | 推計レコード数 |
|------------|-------------|
| 直近1年間（2024年4月以降） | 約8,000〜9,000件 |
| 直近3年間 | 約30,000〜40,000件 |
| 直近5年間（制度改正前含む） | 約60,000〜80,000件 |

#### 合計推計

| シナリオ | JQUANTS | EDINET | 合計 |
|---------|---------|--------|------|
| JQUANTS全件 + EDINET 1年 | 128,000 | 9,000 | **約137,000件** |
| JQUANTS全件 + EDINET 3年 | 128,000 | 35,000 | **約163,000件** |
| JQUANTS全件 + EDINET 5年 | 128,000 | 70,000 | **約198,000件** |

### 3. financial_values（財務数値）

**登録元JOB**: `ImportJquantsFinancialDataJob`, `ImportEdinetDocumentsJob`

ユニーク制約: `(company_id, fiscal_year_end, scope, period_type)`

`financial_reports` とは異なり、同一企業・同一決算期のJQUANTS由来とEDINET由来のデータは **同一レコードにマージ** される。

#### 連結（consolidated）

| 項目 | 計算 | 件数 |
|-----|------|------|
| 1社あたりの年間データ | 4期（通期 + Q1 + Q2 + Q3） | 4 |
| 対象企業数 | 約4,000社 | 4,000 |
| 年間合計 | 4,000 × 4 | 16,000/年 |
| 8年分 | 16,000 × 8 | **約128,000件** |

#### 個別（non_consolidated）

JQUANTSの個別決算データ（NC*フィールド）は全企業が提出するわけではない。連結子会社を持たない企業は連結財務諸表を作成しないため、個別のみの場合もある。

| 項目 | 計算 | 件数 |
|-----|------|------|
| 個別データ提出率 | 約70〜80%の企業 | - |
| 年間合計 | 約3,000 × 4 | 12,000/年 |
| 8年分 | 12,000 × 8 | **約96,000件** |

#### 合計推計

| シナリオ | 連結 | 個別 | 合計 |
|---------|------|------|------|
| JQUANTS 8年分 | 128,000 | 96,000 | **約224,000件** |
| 最小推計（企業数・期間が少なめ） | 100,000 | 70,000 | **約170,000件** |
| 最大推計（企業数・期間が多め） | 150,000 | 110,000 | **約260,000件** |

### 4. financial_metrics（分析指標）

**登録元JOB**: `CalculateFinancialMetricsJob`

`financial_values` と1:1の関係。全ての `financial_values` レコードに対して1つの `financial_metrics` レコードが作成される。

| シナリオ | 推計レコード数 |
|---------|-------------|
| 標準推計 | **約224,000件**（financial_valuesと同数） |
| 最小推計 | 約170,000件 |
| 最大推計 | 約260,000件 |

### 5. daily_quotes（株価四本値）

**登録元JOB**: `ImportDailyQuotesJob`

ユニーク制約: `(company_id, traded_on)`

このテーブルが最大のデータ量を持つ。

#### fullモード（全件取得）

`from_date` デフォルト: `2020-01-01`

| 項目 | 計算 | 件数 |
|-----|------|------|
| 上場企業数 | 約4,000社 | 4,000 |
| 年間取引日数 | 約245日 | 245 |
| 1社あたり年間レコード数 | 245 | 245 |
| 1年分の全企業合計 | 4,000 × 245 | **980,000/年** |

| データ取得期間 | 年数 | 推計レコード数 |
|-------------|------|-------------|
| 2020年〜2026年（デフォルト） | 約6年 | **約5,880,000件** |
| 2017年〜2026年 | 約9年 | **約8,820,000件** |
| 2012年〜2026年（最大） | 約14年 | **約13,720,000件** |

注意事項:
- 全企業が全取引日に取引されるわけではないため、実際のレコード数は上記推計より少なくなる可能性がある
- 上場廃止企業は途中までのデータしか存在しない
- 新規上場企業は上場日以降のデータしか存在しない
- 実際のレコード数は推計の **80〜90%** 程度と見込まれる

#### 差分更新モード（日次実行）

| 項目 | 計算 | 件数 |
|-----|------|------|
| 1日あたりの新規レコード | 約3,500〜4,000 | 約4,000/日 |
| 1ヶ月あたり（約21営業日） | 4,000 × 21 | 約84,000/月 |

### 6. application_properties（アプリケーションメタデータ）

**登録元JOB**: `SyncCompaniesJob`, `ImportJquantsFinancialDataJob`, `ImportEdinetDocumentsJob`, `ImportDailyQuotesJob`

`kind` カラムにユニーク制約があり、定義済みのkindは3種類:

| kind | 用途 |
|-----|------|
| default (0) | デフォルト |
| edinet_sync (1) | EDINET同期状態管理 |
| jquants_sync (2) | JQUANTS同期状態管理 |

**推計レコード数: 2〜3件**（固定）

## データ量の考察

### SQLiteにおける容量推計

SQLiteでは、各レコードのカラム数・データ型によってファイルサイズが変わる。目安として:

| テーブル | 1レコード概算サイズ | 標準シナリオ推計容量 |
|---------|-----------------|-------------------|
| companies | 約500バイト | 約2MB |
| financial_reports | 約300バイト | 約41MB |
| financial_values | 約800バイト（JSON含む） | 約179MB |
| financial_metrics | 約600バイト（JSON含む） | 約134MB |
| daily_quotes | 約200バイト | 約1,060MB |
| application_properties | 約200バイト | 約1KB |
| **合計（インデックス含む）** | - | **約1.4〜2.0GB** |

`daily_quotes` が全体の約70〜75%を占める最大テーブルとなる。SQLiteの実用上限（数十GB程度）に対して十分余裕がある。

### JOB実行時間の推計

| JOB | 主なボトルネック | 推計実行時間 |
|----|---------------|------------|
| SyncCompaniesJob | API 1回 + DB upsert 4,000件 | 数十秒 |
| ImportJquantsFinancialDataJob (full) | 4,000社 × API呼び出し | 数時間（レート制限依存） |
| ImportEdinetDocumentsJob (1年分) | 約8,000書類 × 4秒sleep | **約9〜10時間** |
| ImportDailyQuotesJob (full) | 4,000社 × 1秒sleep | **約1.5〜2時間**（API応答時間含む） |
| CalculateFinancialMetricsJob | 224,000件の算出 + DB読み書き | 数分〜数十分 |

特に `ImportEdinetDocumentsJob` は `SLEEP_BETWEEN_DOCS = 4` 秒のレート制限対応により、大量のドキュメントを処理する場合に非常に長い実行時間が見込まれる。

### 日次運用時の増分レコード数

定常運用（日次差分更新）における1日あたりの増分:

| テーブル | 1日あたり増分（概算） |
|---------|-------------------|
| companies | 0〜数件（新規上場・上場廃止時のみ） |
| financial_reports | 0〜100件（決算発表集中月は多い） |
| financial_values | 0〜100件（上記と連動） |
| financial_metrics | 0〜100件（上記と連動） |
| daily_quotes | 約3,500〜4,000件 |
| application_properties | 0件（更新のみ） |

決算発表集中月（6月・8月・11月・2月）は `financial_reports` / `financial_values` の日次増分が大幅に増加する。

## 補足: データソースの特性比較

| 特性 | JQUANTS | EDINET |
|-----|---------|--------|
| データ形式 | 構造化JSON（フィールド固定） | XBRL（要素名が企業により変動） |
| カバー範囲 | TSE上場銘柄 | 有報提出義務のある全企業 |
| 更新頻度 | 決算短信ベース（速報性高い） | 法定開示ベース（正確性高い） |
| 財務データの粒度 | P/L・B/S・C/F主要項目 + 予想 | P/L・B/S・C/F詳細項目（拡張B/S含む） |
| レート制限 | プラン依存（Free: 5req/min〜Premium: 500req/min） | 書類一覧: 1回/分、書類取得: 3〜5秒間隔 |
| 本システムでの位置づけ | 主データソース | 補完データソース（拡張B/S項目の補完） |
